## 一、项目背景

### 1.1 社团活动形态

我们是一个大学内的**商业模式教学社团**，主要教学和活动内容包括两大模块：

| 模块 | 内容 | 形式 |
|------|------|------|
| **讲座模块** | 邀请嘉宾和教授开展讲座 | 单向输出，被动学习 |
| **小组研究模块** | 学员分组完成课题研究、讨论和展示 | 主动实践，协作学习 |

本产品聚焦于**小组研究模块**的AI赋能。

---

## 二、问题分析

### 2.1 现实痛点

在小组讨论场景中，存在**两类典型的参与者失衡**：

| 类型 | 表现 | 问题 | 影响 |
|------|------|------|------|
| 高参与/低能力 | 表达积极、发言频繁 | 逻辑弱、观点浅 | 占据讨论空间 |
| 低参与/高能力（I型） | 有思考深度，但不愿发言 | 能力未被利用 | 优质观点流失 |

### 2.2 核心矛盾

**不是"让大家多说"，而是：让每个人在最合适的位置参与，并产生有效输出。**

---

## 三、产品定位

### 3.1 产品定位

**AI Project Coach / AI Discussion Facilitator**

核心能力：
- 帮助小组从模糊课题变成明确研究计划
- 根据成员背景做分工、推进和检查
- 在多轮讨论中持续沉淀项目记忆
- 自动维护项目文档、会议纪要、任务清单
- 在会议中引导讨论，避免低效发散

### 3.2 目标用户

| 用户角色 | 核心诉求 |
|----------|----------|
| 教师 | 全局掌控、进度跟踪、成果验收 |
| 组长 | 高效组织、任务分配、进度推进 |
| 组员 | 个人成长、轻松参与、获得反馈 |

## 五、核心架构设计

### 5.1 多Agent协作架构（重要设计）

**设计理念**：将一个"全能Agent"拆分为多个**专业角色**，前台用户感知到一个统一的Agent，但后台由多个专业角色协作完成复杂任务。

#### 5.1.1 Agent角色划分

| 角色 | 职责 | 能力特点 |
|------|------|----------|
| **主控Agent（Coordinator）** | 对外交互、意图识别、任务分发 | 理解力强、会话流畅 |
| **研究员（Researcher）** | 市场调研、信息搜集、数据分析 | 知识广度、检索能力 |
| **产品设计师（Designer）** | 产品设计、方案输出、文档撰写 | 结构化思维、专业表达 |
| **会议主持（Facilitator）** | 会议引导、节奏控制、纪要生成 | 组织能力、总结能力 |
| **评估师（Evaluator）** | 发言评估、质量反馈、能力分析 | 批判思维、洞察力 |

#### 5.1.2 协作流程

```
用户消息
    │
    ▼
┌─────────────────┐
│  主控Agent      │ ← 接收用户输入，识别意图
│  (Coordinator)  │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│                        任务分发                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  "帮我调研市场"     →  研究员                                    │
│  "设计产品方案"     →  产品设计师                                 │
│  "开个会"          →  会议主持                                   │
│  "评估一下发言"     →  评估师                                    │
│  通用对话          →  主控Agent直接回复                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│  结果整合       │ ← 主控Agent整合各角色输出
└────────┬────────┘
         │
         ▼
用户回复
```

#### 5.1.3 角色协作示例

**场景**：用户说"帮我调研一下新能源汽车市场，然后设计一个校园共享充电桩的产品方案"

```
Step 1: 主控Agent识别到复合任务
        → 拆分为：调研任务 + 设计任务

Step 2: 并行调用
        → 研究员：开始调研新能源汽车市场
        → 产品设计师：准备产品设计方案框架

Step 3: 研究员输出调研结果
        → 主控Agent接收

Step 4: 产品设计师基于调研结果
        → 生成产品设计方案

Step 5: 主控Agent整合输出
        → 给用户完整的调研报告+产品方案
```

## 七、技术架构

### 7.1 技术栈选择

| 层级 | 方案 | 说明 |
|------|------|------|
| **前端** | Streamlit | 快速开发，3天出Demo |
| **后端** | FastAPI | 轻量API服务 |
| **数据库** | SQLite + JSON | 免部署 |
| **LLM** | OpenAI GPT-4o | Demo主力模型 |

### 7.2 短期 vs 长期技术路线

| 阶段 | 技术路线 | 说明 |
|------|----------|------|
| **Demo V1.0（3天）** | Streamlit自研 | 快速验证核心流程 |
| **MVP V1.5（2周）** | Streamlit + 优化 | 完善功能、用户系统 |
| **产品 V2.0（1个月）** | **Dify + Streamlit** | 可视化编排、多Agent协作 |
| **高级版** | LangGraph自研 | 完整的多Agent工作流 |

### 7.3 Dify集成规划（长期）

**为什么需要Dify**：
- 单Agent难以处理复杂任务编排
- 多Agent协作需要可靠的工作流框架
- Dify提供可视化编排，降低开发成本

**集成方案**：
```
┌─────────────────┐
│  Streamlit前端  │  ← 用户界面
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   Dify API      │  ← Agent引擎+Workflow
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Mem0/Letta     │  ← Memory管理
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  OpenAI API     │  ← LLM能力
└─────────────────┘
```
